Relay: Unbreakable Coding Rules
v0.1 — May 2026
These rules apply to every line of code written for Relay, whether by a human or an AI assistant. They are non-negotiable. When in doubt, refer back here before writing anything.


Section A — Foundation rules
Rule
	Name & Description
	R1
	One file, one responsibility
Every module does exactly one thing. context_broker.py signs envelopes. validator.py checks handoffs. snapshot.py persists state. If you are unsure which file a function belongs in, the function is probably doing too many things.
	R2
	No shared mutable state
Agents never write to a shared object. The context envelope is read-only from the moment it is created. Use Python dataclasses with frozen=True or Pydantic models with model_config frozen=True. If you need to update state, create a new envelope.
	R3
	Type everything
Every function must have full type annotations on parameters and return values. No bare Any. No missing return types. Use mypy --strict in CI. If mypy complains, fix the code, not the config.
	R4
	Errors are values, not surprises
Use Result types (returns a value or an error object) instead of raising exceptions for expected failure cases. Only raise exceptions for programmer errors and truly unrecoverable states. Never swallow an exception with a bare except pass.
	

Section B — Testing rules
Rule
	Name & Description
	R5
	Every public function has a test
No exceptions. If a function is public and has no test, it does not exist from a quality standpoint. Tests live in tests/ mirroring the src/ structure. Use pytest. Coverage must stay above 80%.
	R6
	Tests are readable sentences
Test names must describe behaviour, not implementation. GOOD: test_validator_rolls_back_on_contradiction. BAD: test_validator_1. If you cannot write a plain-English description of what a test checks, the test is not ready.
	R7
	No network calls in unit tests
Unit tests must run offline. Mock all LLM calls, HTTP requests, and file I/O in unit tests. Integration tests that hit real APIs live in tests/integration/ and are opt-in only.
	

Section C — AI-assisted coding rules
Rule
	Name & Description
	R8
	Never paste AI output directly
Every AI-generated code block must be read line by line before it enters the codebase. You are responsible for understanding every line you commit, regardless of who or what wrote it. If you do not understand a line, do not commit it.
	R9
	AI writes implementations, humans write interfaces
Design the function signatures, type contracts, and error behaviours yourself before asking AI to fill in the body. AI is excellent at implementation details. It is unreliable at API design. Never let it decide what a public function should look like.
	R10
	Ask AI to break its own output
After AI generates a function, ask it: what inputs would make this function fail or behave unexpectedly? Use those answers to write edge case tests before the code ships. This is the single highest-leverage use of AI in a dev workflow.
	R11
	One task per AI prompt
Give AI one clearly scoped task at a time. Do not ask it to write a module, its tests, and its documentation in one prompt. The output quality degrades dramatically. Scope each prompt to a single function or a single test file.
	

Section D — Commit and review rules
Rule
	Name & Description
	R12
	Commits are atomic and describable
Every commit must do exactly one thing and its message must say what and why, not how. GOOD: fix: rollback triggered on empty diff instead of contradiction. BAD: fixed stuff. If you cannot write a one-sentence commit message, split the commit.
	R13
	No commented-out code
Dead code is not a safety net, it is noise. If code is not needed, delete it. Git history is the safety net. Commented-out code in a PR is grounds for rejection.
	R14
	Documentation is code
Every public function needs a one-line docstring explaining what it does, not how. Every module needs a three-line module docstring: what it does, what it owns, what it does not do. Write these before writing the implementation.
	

Quick Reference
Before writing any code, ask:
* Which layer does this belong to?
* Does this module already exist?
* What is the type signature?
* What is the test for the happy path?
* What is the test for the failure path?
Before committing, check:
* mypy passes with no errors
* All new public functions have tests
* No commented-out code
* Commit message follows the format
When using AI, remember:
* You designed the interface, AI filled the body
* You read every line before committing
* You asked AI how to break its own output
* You wrote the tests yourself
