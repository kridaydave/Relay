## Baseline

Relay is a middleware library with one job: safe context passing between AI agents via signed envelopes, snapshots, and deterministic rollback. Design Principle 2: **Determinism over cleverness.** The codebase is at v0.4.0 with the parallel fork-join layer active.

**The Law:**
- Rule 1.2: Layered imports — strict hierarchy, lower layers never import upper
- Rule 2.1: `mypy --strict`, zero suppressions
- Rule 3.3: Error codes are a public API — enum only, no strings
- Rule 7.5: Every `Result`-returning function has tests for each distinct `Failure` code
- Rule 8.2: Docstrings stay accurate after every behaviour change
- Design Principle 1: One job only

### RULE-05 — `RecencySlicePacker` and `RelevanceSlicePacker` use `continue` instead of `break` — non-recency behaviour (Rule 4.1)

**File:** `src/relay/slicer/packers.py:73,144`
**Severity:** Low

```python
for key in sorted_keys:        # newest-first
    if manifest.max_tokens is not None and used_tokens + section_tokens > manifest.max_tokens:
```

When a section exceeds the remaining budget, `continue` tries the next (older) section rather than stopping. ### MED-01 — `execute_parallel_step` two-step state commit creates a concurrency window

**File:** `src/relay/core_pipeline.py:503-521`
**Severity:** Medium

```python
commit_result = self.execute_step_with_manifest(merged_result.value, manifest=None)
# Lock released — current envelope is now unsigned/metadata-free

# ... metadata attached, re-signed ...

save_result = self._snapshot_store.save_snapshot(signed)  # outside lock
with self._state.transaction():
    self._state.set_current(signed)                        # second lock acquisition
```

Between `execute_step_with_manifest` returning and the final `set_current(signed)`, any `get_current_envelope()` call returns the intermediate unsigned envelope — fork metadata not attached, signature invalid. The v0.4 plan's concurrency model addresses forks during adapter I/O, but not this commit gap.

### BUG-M-03 — `core_pipeline.py:254-262` — State corruption when rollback snapshot fails

**File:** `src/relay/core_pipeline.py:254-262`

```python
self._state.register_snapshot(current_envelope.step, save_result.value)
self._state.archive_and_set(new_envelope)  # STATE IS NOW CORRUPTED
return self._rollback_on_contradiction(new_envelope, validation_result.value)
```

`archive_and_set(new_envelope)` updates the pipeline state to the bad envelope BEFORE `_rollback_on_contradiction` is called. If `_rollback_handler.restore_to_previous` fails (e.g., snapshot deleted between registration and load), the state is left pointing at the contradictory envelope.

**Severity:** High (as stated for the failure scenario).

> ⚠️ **PARTIALLY VERIFIED — Eric (S.Dev):** Structurally correct. After `archive_and_set(new_envelope)`, if `restore_to_previous` returns `Failure`, `set_current` is not called and `_current_envelope` remains `new_envelope` (the contradictory one). In normal operation the failure scenario is narrow: the snapshot was registered immediately before rollback is called, so `NO_SNAPSHOT_REGISTERED` is unlikely. The real risk is OS-level failure between `register_snapshot` and `load_snapshot` (file deleted, permission revoked). Severity is Medium in practice, High in principle. The state invariant should be hardened regardless.
>
> *See also: Eric MED-01 (same root cause, different manifestation).*

### MED-M-04 — `parallel/join.py:48-52,88-91` — Raises `ValueError` instead of returning `Failure`

Both `_apply_union` and `_apply_vote` raise `ValueError` when `agent_output is None` on a `success=True` result. Rule 3.1 mandates returning `Failure` for operational errors.

**Severity:** Low (invariant violation should never occur in practice).

> ✅ **VERIFIED — Eric (S.Dev):** Confirmed. The guard is:
> ```python
> raise ValueError("ForkResult.success=True but agent_output is None — invariant violated")
> ```
> Rule 3.1 says raise `ValueError` for programmer errors (invariant violations) and return `Failure` for operational errors. A `success=True` ForkResult with `agent_output=None` IS a programmer error (the invariant is broken at ForkResult construction time), so `ValueError` is actually correct per the rule. However, the `zero-exception policy` interpretation is that nothing in a Result-returning call chain should ever raise to callers. Given these functions are async pipeline internals, a `ValueError` here crashes `asyncio.gather` — callers cannot distinguish it from an `Exception` in the adapter. The correct fix is to convert to `Failure(UNKNOWN_ERROR)` with a descriptive reason, consistent with how adapter exceptions are handled in `_run_single_fork:74-87`.

### MED-M-05 — `core_pipeline.py:212-216` — No validation of negative `budget_used`

`_check_budget` reads `current_envelope.token_budget_used` without checking for negative values.

> ⚠️ **PARTIALLY VERIFIED — Eric (S.Dev):** `ContextEnvelope.__post_init__` rejects `token_budget_used < 0` at construction time (see BUG-M-01 false positive note). So a negative value cannot enter via the normal construction path. However, `_dict_to_envelope` could produce a negative value if a corrupted snapshot contains a negative integer (since `_require_int` only checks type, not sign). In that scenario, the negative budget would pass `_check_budget` silently. Narrow but valid.
